# 阶段三总结报告：CrewAI 多智能体协作

## 1. 核心架构思想：声明式的“模拟公司”
与 LangGraph 必须手动用连线（Edges）把节点（Nodes）串起来的“命令式编程”不同，CrewAI 采用的是**“声明式（Declarative）编排”**。你只需要提供资源和边界，具体的推导由框架内部运转。
它将多 Agent 抽象为现实世界中的企业组织架构模式。

## 2. 组件职责（组件映射表）
*   **Agent（智能体/员工）**：核心属性是大模型（`llm`）、角色（`role`）、目标（`goal`）和背景（`backstory`）。负责提供“思考和干活”的专业脑力。
*   **Task（任务项）**：包含任务描述和预期输出格式。它将被绑定到具体的 Agent 或交给 Manager 分派。
*   **Crew（团队/公司大群）**：容器组件。把所有参与的 Agent 和 Task 打包在一起启动（`kickoff`）的执行引擎。
*   **Process（流程/组织架构）**：
    *   **Sequential（流水线模式）**：前一个人的 Task 输出结果，自动变成后一个人的上下文环境。
    *   **Hierarchical（层级包裹模式）**：系统自动创建一个基于 LLM 的 Manager，它像黑盒一样去根据任务大纲向员工派活和验收。
*   **Flows（事件流控）**：利用 `@start`、`@listen`、`@router` 处理宏观业务流，这弥补了纯 Crew 结构无法插入硬性业务红线的缺陷。

## 3. 数据流与状态变化
*   **在 Crew 内部**：状态传递是隐式的。任务 A 的 `expected_output` 会被自动拼接成 Prompt 注入给任务 B。
*   **在 Flow 层面**：状态是显式的。通过定义一个继承自 `BaseModel` 的类（如 `ResearchState`），各个节点通过修改 `self.state` 来实现数据的全局读写和状态更新。

## 4. 最佳应用场景
当你需要处理**目标相对明确，但需要多个专业视角交叉碰撞**的任务时（如：让 AI 调研竞品资料+让 AI 排版+让 AI 校对），用 CrewAI 能省去你写海量底层代码的麻烦。
